# Service Portal Patterns — Fluent SDK

Patterns for building ServiceNow Service Portal portals and widgets using the
ServiceNow SDK Fluent API. Service Portal uses **AngularJS-based widgets** and
classic `sp_*` tables — this is distinct from `UiPage` (which is React-based).

**Official API docs:** https://www.servicenow.com/docs/r/application-development/servicenow-sdk/fluent-service-portal-api.html
**Official example repo:** https://github.com/ServiceNow/sdk-examples/tree/626cd89b9df8386d518361ae2647f46abd356360/service-portal-sample

---

## Key Distinction: Service Portal vs UI Page

| | `SPWidget` / Service Portal | `UiPage` |
|---|---|---|
| Import | `SPWidget` from `@servicenow/sdk/core` | `UiPage` from `@servicenow/sdk/core` |
| Framework | AngularJS | React + TypeScript |
| Server data | `data` object in `server.js` | REST API via `fetch()` |
| Auth | `$sp`, `$http` AngularJS services | `window.g_ck` token |
| Styling | SCSS (scoped per widget) | Tailwind / CSS modules |
| Page layout | `sp_container` / `sp_row` / `sp_column` records | React Router / JSX |
| URL | `/<instance>/<portal_url_suffix>.do` | `/<instance>/<endpoint>.do` |
| Use case | Classic ServiceNow Service Portal | Custom modern portals |

```
service-portal-app/
├── now.config.json
├── package.json
├── assets/
│   ├── logo.svg
│   └── icon.jpg
└── src/fluent/
    ├── portal/
    │   └── sp_portal.now.ts              ← portal record (sp_portal table)
    ├── page/
    │   ├── sp_page_<sys_id>.now.ts       ← page record (sp_page table)
    │   ├── sp_container_<sys_id>.now.ts  ← layout container
    │   ├── sp_row_<sys_id>.now.ts        ← row within container
    │   ├── sp_column_<sys_id>.now.ts     ← column within row
    │   └── sp_instance_<sys_id>.now.ts   ← widget instance placed on page
    └── widgets/
        ├── my-widget.now.ts              ← SPWidget definition
        ├── my-widget.client.js           ← AngularJS controller
        ├── my-widget.server.js           ← server-side GlideRecord script
        ├── my-widget.html                ← AngularJS HTML template
        └── my-widget.scss                ← scoped styles
```

> **Note:** The `sp_page`, `sp_container`, `sp_row`, `sp_column`, and `sp_instance`
> records use their `sys_id` in the filename. These are typically generated by
> `now-sdk transform` when pulling from an existing instance, or assigned manually
> as `ffffffffffffffffffffffffffffffff`-style IDs.

## 1. Portal Record (`sp_portal.now.ts`)

Creates the top-level portal accessible at `/<instance>/<url_suffix>.do`.

```typescript
import { Record } from '@servicenow/sdk/core'

Record({
    $id: Now.ID['my-portal'],
    table: 'sp_portal',
    data: {
        title: 'My Customer Portal',
        url_suffix: 'myportal',
        logo: Now.attach('../../../assets/logo.svg'),
        icon: Now.attach('../../../assets/icon.jpg'),
    },
})
```

**Key fields:**
- `url_suffix` — the URL path, e.g. `myportal` → `/<instance>/myportal.do`
- `logo` / `icon` — use `Now.attach()` to reference local asset files
- `homepage` — `sys_id` of the default `sp_page` record (optional)
- `theme` — `sys_id` of an `sp_theme` record (optional)

## 2. Widget Definition (`my-widget.now.ts`)

The central Fluent definition that stitches together all widget files.

```typescript
import { SPWidget } from '@servicenow/sdk/core'

// sys_ids of JavaScript library dependencies (Chart.js, etc.)
const CHARTJS = 'a7a8754347011200ba13a5554ee4905c'
// sys_ids of Angular provider dependencies
const SP_ELLIPSIS_TOOLTIP = '1d2b40e07323201081d3738234f6a714'

SPWidget({
    $id: Now.ID['my-widget'],
    name: 'My Widget',           // display name in Studio
    id: 'my-widget',             // unique widget ID used in sp_instance
    clientScript: Now.include('my-widget.client.js'),
    serverScript: Now.include('my-widget.server.js'),
    htmlTemplate: Now.include('my-widget.html'),
    customCss: Now.include('my-widget.scss'),
    demoData: {
        data: {
            items: [],           // mirrors what server script puts in `data`
        },
    },
    hasPreview: true,
    linkScript: `function link(scope, element, attrs, controller) {
}`,
    dependencies: [CHARTJS],          // optional: JS lib sys_ids
    angularProviders: [SP_ELLIPSIS_TOOLTIP],  // optional: Angular provider sys_ids
})
```

**Key rules:**
- `id` must be unique across the portal — used to reference the widget in `sp_instance`
- `Now.include('filename.js')` paths are **relative to the `.now.ts` file**
- `demoData` mirrors the shape of `data` your server script produces — used in widget preview
- `dependencies` and `angularProviders` are arrays of `sys_id` strings from the instance
- `linkScript` is an AngularJS link function string — include even if empty

## 3. Server Script (`my-widget.server.js`)

Runs on the server. Populates the `data` object which is passed to the client controller.

```javascript
(function () {
    // Use GlideQuery (modern) or GlideRecord
    data.cases = new global.GlideQuery('sn_customerservice_case')
        .where('opened_by', gs.getUserID())
        .orderByDesc('opened_at')
        .limit(10)
        .select('number', 'short_description', 'state')
        .toArray(10);

    // Or classic GlideRecord:
    // var gr = new GlideRecord('incident');
    // gr.addActiveQuery();
    // gr.query();
    // data.incidents = [];
    // while (gr.next()) {
    //     data.incidents.push({ number: gr.getValue('number') });
    // }
})();
```

**Key rules:**
- Wrap everything in an IIFE `(function() { ... })();`
- Write to `data.<field>` — this object is serialized and sent to the client
- Use `gs.getUserID()` / `gs.getUserName()` for current user context
- `$sp` object is also available for Service Portal utilities (e.g. `$sp.getWidget()`)
- GlideQuery returns plain JS objects — preferred over GlideRecord for clean `data` shape

## 4. Client Script (`my-widget.client.js`)

AngularJS controller. Receives `data` from server via `c.data`.

```javascript
api.controller = function ($scope, $http) {
    var c = this;
    // c.data is populated by the server script
    $scope.items = c.data.items;

    // Call a server-side function (spUtil / $http)
    $scope.refresh = function () {
        $http.get('/api/now/table/incident?sysparm_limit=5').then(function (response) {
            $scope.items = response.data.result;
        });
    };
};
```

**Key rules:**
- Always assigned to `api.controller`
- `var c = this;` — `c.data` is the server-populated data object
- Inject AngularJS services in the function signature: `$scope`, `$http`, `$location`, etc.
- `spUtil` is available for SP-specific utilities

## 5. HTML Template (`my-widget.html`)

AngularJS template bound to the client controller's `$scope`.

```html
<div class="widget-wrapper">
  <h3 sp-ellipsis-tooltip sp-ellipsis-tooltip-title="My Widget">My Widget</h3>
  <ul class="list-unstyled">
    <li ng-repeat="item in items track by $index">
      <a href="{{item.link}}">{{item.label}}</a>
    </li>
  </ul>
  <p ng-if="items.length === 0" class="text-muted">No items found.</p>
</div>
```

**Key rules:**
- Standard AngularJS directives: `ng-repeat`, `ng-if`, `ng-click`, `ng-model`, etc.
- Bootstrap 3 classes are available (Service Portal ships with Bootstrap 3)
- `sp-ellipsis-tooltip` is a common SP directive for truncated text with tooltip

## 6. SCSS (`my-widget.scss`)

Scoped to the widget — styles here won't leak outside the widget boundary.

```scss
.widget-wrapper {
  padding: 16px;

  h3 {
    font-size: 18px;
    color: $sp-widget-header-color;  // SP SCSS variable
  }

  .list-unstyled li {
    padding: 8px 0;
    border-bottom: 1px solid $sp-divider-color;
  }
}
```

**Common SP SCSS variables:** `$sp-widget-header-color`, `$sp-divider-color`,
`$sp-brand-primary`, `$sp-nav-bg-color`

## 7. Page Layout Records

These records define where widgets appear on a page. Typically generated via
`now-sdk transform` from an existing instance, then committed to source control.

### Page (`sp_page_<sys_id>.now.ts`)
```typescript
import { Record } from '@servicenow/sdk/core'

Record({
    $id: Now.ID['sample-page'],
    table: 'sp_page',
    data: {
        id: 'sample_page',
        title: 'sample-page',
        category: 'custom',
        draft: false,
        internal: false,
        public: false,
    },
})
```

### Container → Row → Column → Instance (hierarchy)
```
sp_page
  └── sp_container   (full-width section)
        └── sp_row
              └── sp_column  (defines grid width, e.g. col-md-12)
                    └── sp_instance  (places a specific widget here)
```

### Widget Instance (`sp_instance_<sys_id>.now.ts`)
```typescript
import { Record } from '@servicenow/sdk/core'

Record({
    $id: Now.ID['my-widget-instance'],
    table: 'sp_instance',
    data: {
        widget: '<widget_sys_id>',  // sys_id of the sp_widget record
        column: '<column_sys_id>',  // sys_id of the sp_column record
        order: 1,
        async: false,
        size_md: 'col-md-12',
        size_sm: 'col-sm-12',
        size_xs: 'col-xs-12',
    },
})
```

## 9. Build & Deploy

```bash
# Compile Fluent → metadata XML
now-sdk build

# Deploy to instance
now-sdk install --auth <alias>

# Pull existing portal config from instance → Fluent files
now-sdk transform --auth <alias>
```

After `transform`, generated files land in `src/fluent/generated/` — migrate
them to the proper directory structure if you want to edit them.

## 10. Troubleshooting

| Problem | Solution |
|---|---|
| `SPWidget is not exported from '@servicenow/sdk/core'` | Update SDK: `npm install @servicenow/sdk@latest` |
| Widget not showing on page | Check `sp_instance` column/widget `sys_id` references are correct |
| `data` is empty on client | Ensure server script writes to `data.<field>`, not a local variable |
| `Now.include()` file not found | Path must be relative to the `.now.ts` file, not the project root |
| `Now.attach()` file not found | Path must be relative to the `.now.ts` file |
| Portal returns 404 | Check `url_suffix` in `sp_portal` record matches the URL you're hitting |
| `transform` overwrites manual changes | Move generated files out of `src/fluent/generated/` before editing |
| AngularJS `$http` not injected | Add `$http` to `api.controller = function($scope, $http)` signature |

## 11. Lessons Learned from Real Deployments

### CSS / SCSS

**❌ CSS `display: grid` does NOT work in Service Portal**

SP runs Bootstrap 3 — its base styles and the browser-render environment don't
reliably apply CSS Grid Layout. Widgets using `display: grid` will fall back to
stacked full-width blocks.

**✅ Always use Bootstrap 3 grid classes in HTML templates instead:**

```html
<!-- ❌ Wrong — CSS grid, will break in SP -->
<div class="my-grid">
  <a ng-repeat="item in items" class="my-card">...</a>
</div>

<!-- ✅ Correct — Bootstrap row/col inside the template -->
<div class="row">
  <div class="col-md-4 col-sm-6" ng-repeat="item in items">
    <a class="my-card" href="{{item.href}}">...</a>
  </div>
</div>
```

Common column layouts:
- 3-column grid: `col-md-4 col-sm-6 col-xs-12`
- 4-column grid: `col-md-3 col-sm-6 col-xs-12`
- 2-column grid: `col-md-6 col-sm-12`

---

**❌ Nested SCSS (`&__modifier` BEM) is unreliable in SP**

The SP SDK's SCSS compiler sometimes fails to apply deeply nested BEM-style
selectors within widget `customCss`. Styles silently don't apply.

**✅ Always use flat class selectors — no nesting, no `&` references:**

```scss
/* ❌ Wrong — nested BEM, may not apply */
.my-widget {
  &__card { background: #fff; }
  &__card:hover { background: #f0f0f0; }
  &__title { font-weight: bold; }
}

/* ✅ Correct — flat selectors */
.my-card { background: #fff; }
.my-card:hover { background: #f0f0f0; }
.my-card-title { font-weight: bold; }
```

The official SDK sample widget (`sample-widget.scss`) uses only flat selectors —
follow that pattern.

---

**❌ Widget CSS classes are scoped — shared utility classes must be re-declared per widget**

Each widget's SCSS is isolated. If two widgets share a class like `.cp-inner`
(a centering wrapper), each widget must define it in its own SCSS file.
You cannot rely on another widget's CSS being available.

```scss
/* Add this in every widget that uses .cp-inner */
.cp-inner {
  max-width: 1172px;
  margin: 0 auto;
  padding: 0 24px;
}
```

---

### Widget Instance Layout

**❌ `async: true` on `sp_instance` causes silent rendering failures**

When a widget instance is set to `async: true`, it loads via AJAX after the
initial page render. If the widget's server script throws any error, the widget
fails silently — nothing renders and no visible error is shown.

**✅ Use `async: false` for all widgets until confirmed stable:**

```typescript
Record({
    $id: Now.ID['my-inst'],
    table: 'sp_instance',
    data: {
        column: COLUMN_SYS_ID,
        widget: WIDGET_SYS_ID,
        order: 100,
        async: false,  // ✅ synchronous — errors surface visibly
    },
})
```

Only switch to `async: true` once the widget is working correctly, and only for
widgets with slow server-side queries that should not block the initial page load.

### Scoped App Server Script Restrictions

These GlideRecord/GlideSystem methods are **blocked in scoped apps** — use the alternatives:

| ❌ Blocked | ✅ Scoped-safe alternative |
|---|---|
| `user.getFullName()` | `gs.getUserDisplayName()` |
| `new GlideDateTime().getYear()` | `new GlideDateTime().getYearLocalTime()` |
| `gr.forEach(fn)` | `while (gr.next()) { ... }` |
| `addAggregateQuery('COUNT')` | `addAggregate('COUNT')` |
| `sn_customerservice_case` table | `incident` table (CSM plugin required for case) |

**GlideAggregate IN query** — use `addQuery('field', 'IN', 'val1,val2')` (3-arg form):
```javascript
var gr = new GlideAggregate('incident');
gr.addQuery('state', 'IN', '1,2');  // ✅ correct
gr.addAggregate('COUNT');
gr.query();
```
